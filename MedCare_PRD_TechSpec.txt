1. Product Overview
1.1 Vision
MedCare is a unified health companion app that helps users follow through on medical advice — whether they received it from an in-app teleconsultation or from an outside doctor or hospital. The core value proposition is turning a prescription or discharge summary into a structured, trackable care plan with automated reminders.

1.2 The Two Entry Doors
The home screen presents two clear paths. Both feed into the same care plan engine underneath.

Door A — Consult a Doctor
Door B — Upload Prescription / Discharge
For users who need to see a doctor first.
	•	Fever, cough, stomach issues
	•	Minor infections
	•	Chronic condition follow-ups
Phase 2 feature.
For users who already have a plan.
	•	Visited a doctor outside the app
	•	Post-hospitalisation discharge
	•	Post-surgery recovery plans
v1 primary focus.

1.3 Target Users
	•	Urban individuals aged 25–55 managing their own or family health
	•	Caregivers managing medications for elderly parents
	•	Chronic patients (diabetes, hypertension, thyroid) on long-term medication
	•	Post-hospitalisation patients who need structured recovery plans
	•	Parents tracking children's acute illness care plans

2. User Flows & Screens
2.1 Onboarding Flow
	•	Splash screen → App name + tagline ("Your care plan, always with you")
	•	Sign up / Log in — Phone number OTP (Indian market standard)
	•	Profile setup — Name, age, gender, known conditions (optional)
	•	Home screen — Two primary CTAs
	•	Permissions — Notification permission prompt (critical for reminders)

Design Note
Onboarding must be under 60 seconds. Do not ask for health history upfront — this creates friction. Collect it lazily as users create episodes.

2.2 Core Flow: Upload Prescription (v1 Primary)
Step 1 — Capture
	•	User taps 'Upload Prescription' or 'Upload Discharge Summary'
	•	Options: Camera capture (live), Gallery upload, PDF upload
	•	Support multi-page uploads for discharge summaries (up to 10 pages)

Step 2 — AI Extraction
	•	Backend sends image to OCR + AI extraction pipeline
	•	Extracts: Medicine names, dosage, frequency, duration, doctor name, date
	•	For discharge summaries: also extracts tests ordered, lifestyle instructions, follow-up date
	•	Returns structured JSON to the app

Critical Safety Rule
NEVER auto-confirm extracted data. Always route to the manual confirmation screen first. Show original image alongside extracted data. Any field with low confidence score must be flagged with a warning icon. The user must explicitly tap 'Confirm' on every medicine before the plan is created.

Step 3 — User Confirmation Screen
	•	Side-by-side view: original prescription image + extracted data
	•	Each medicine listed as an editable card: name, dose, frequency, duration
	•	User can edit any field directly
	•	Low-confidence extractions highlighted in amber
	•	'Add medicine manually' option always visible
	•	'Confirm and create plan' CTA at the bottom

Step 4 — Plan Created
	•	Episode created with all confirmed medicines and tasks
	•	User lands on the Episode Detail screen (Plan tab active)
	•	Notification permission prompt if not already granted
	•	Onboarding tooltip: 'Your first reminder is set — here is what to expect'

2.4 Family / Caregiver Flow
	•	One account supports multiple profiles (e.g. Self, Mum, Dad, Child)
	•	Each profile has its own episodes, medicines, and reminders
	•	Caregiver view: single dashboard showing all profiles' today schedule
	•	Push notifications go to the account owner's phone for all profiles
	•	Caregiver can mark doses taken on behalf of family member

Scope Note — v1
In v1, all profiles are local to one device/account. Multi-device shared caregiving (e.g. spouse + adult child both managing mum's plan) is a v2 feature requiring real-time sync.

4. API & Backend Architecture
4.1 Architecture Overview
	•	Backend: Node.js (Express) or Python (FastAPI) — REST API
	•	Database: PostgreSQL for relational data
	•	File storage: AWS S3 (prescription images, discharge summaries)
	•	Authentication: Firebase Auth or custom OTP via SMS (Twilio / MSG91 for India)
	•	Push notifications: Firebase Cloud Messaging (FCM) — works for both iOS and Android
	•	AI extraction: OpenAI GPT-4 Vision API or Google Document AI
	•	Hosting: AWS / GCP with auto-scaling

Architecture Principle
In v1, keep the backend simple — a single monolithic REST API is fine. Do not over-engineer with microservices early. Split into services only when you have scaling evidence for a specific bottleneck.

4.3 AI Extraction Pipeline
Flow
	•	App uploads image to S3 via pre-signed URL
	•	App calls POST /episodes/:id/upload with S3 key
	•	Backend sends image to GPT-4 Vision with structured extraction prompt
	•	AI returns JSON: { medicines: [...], tasks: [...], doctor: '', date: '', follow_up: '' }
	•	Backend stores raw AI response alongside confidence scores
	•	Backend returns structured data to app for user confirmation
	•	User confirms / edits and calls POST /episodes/:id/confirm
	•	Backend creates Medicine and Task records from confirmed data

Extraction Prompt Strategy
The prompt sent to GPT-4 Vision should instruct it to return only JSON, with a confidence score per field. Low-confidence fields (below 0.7) must be returned with a flag so the app can highlight them for the user.

Safety Rule (Technical)
Never auto-create Medicine records from AI output without explicit user confirmation. The confirmation step is a hard requirement, not optional UX. Build this as a separate API call so it is impossible to bypass.

4.4 Reminder / Notification System
	•	Reminder schedule generated server-side when medicines are confirmed
	•	Stored as scheduled jobs (e.g. using Bull queue + Redis, or AWS EventBridge)
	•	FCM used to send push notification at scheduled time
	•	Notification payload includes: medicine name, dose, episode title, action buttons
	•	Action buttons (iOS and Android): 'Taken', 'Snooze 30 mins', 'Skip'
	•	User response to notification calls POST /doses/:id/log directly
	•	Missed doses (no response in 2 hours): logged as pending, shown in app

5.2 Tier Enforcement Logic
Free Tier Limits (enforce on backend, not just frontend)
	•	Episode count check: if user has 1 active episode and is Free → block new episode creation, prompt upgrade
	•	Profile check: if Free user tries to add second profile → prompt upgrade
	•	Upload check: if Free user tries to upload prescription photo → prompt upgrade
	•	Grace behaviour: never delete data if user downgrades. Just restrict new actions.

Upgrade Prompt Strategy
	•	Trigger upgrade prompt contextually — not as an interstitial on open
	•	Show prompt when user hits a specific limit, with the exact feature they tried to access named
	•	Offer free trial for Pro (7 days) to reduce friction

5.3 Refill Prediction Logic
	•	Calculate: (stock_count) / (doses_per_day) = days remaining
	•	If days remaining <= refill_reminder_days (default 7): push notification 'Time to refill [Medicine Name]'
	•	User can update stock_count manually or mark 'refilled' from reminder
	•	For chronic medicines: generate monthly refill reminders if stock tracking not enabled

5.4 Adherence Calculation
Adherence % = (doses taken) / (total scheduled doses in period) x 100
	•	Calculated per medicine, per episode, and per period (7d / 30d / full)
	•	'Out of stock' doses counted separately — not as skipped, not as taken
	•	Chronic medicines: rolling 30-day adherence as the primary metric
	•	Acute medicines: full-course adherence shown on episode completion

6. Regulatory & Safety Requirements
6.1 AI Role Boundaries
	•	AI is used ONLY for extraction and organisation of existing prescriptions — never for diagnosis or prescribing
	•	All AI-extracted data must be confirmed by the user before any plan is created
	•	App UI must never phrase AI output as a recommendation or medical advice
	•	Disclaimer must be visible on the prescription confirmation screen: 'Please verify all details with your doctor's prescription. This app does not provide medical advice.'

6.2 Teleconsult Compliance (Phase 2)
	•	All doctors must be Registered Medical Practitioners (RMPs) with valid MCI/NMC registration
	•	Registration number must be displayed on doctor profile and prescriptions
	•	Prescriptions generated in-app must meet Telemedicine Practice Guidelines (MoHFW, 2020)
	•	Controlled substances (Schedule X) cannot be prescribed via teleconsult
	•	Patient consent must be recorded for every teleconsult session

6.3 Data Privacy
	•	All health data encrypted at rest (AES-256) and in transit (TLS 1.3)
	•	Prescription images stored in private S3 bucket with access logging
	•	User data never shared with third parties for advertising
	•	DPDP Act (India, 2023) compliance required — privacy policy and consent flows
	•	Right to data deletion: user can delete account and all associated data

Summary
MedCare is a one app, two entry door product. v1 focuses entirely on prescription upload to care plan to reminders. The data model is designed upfront to support wearables (v2) and teleconsult (v3) without rework. Monetisation is simple: free tier with limits, Pro subscription, and per-consult teleconsult fees later.
